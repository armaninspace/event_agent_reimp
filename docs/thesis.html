From Public Questions To Auditable Event Evidence

A Thesis-Style Technical Report On The Event Agent Research System

Author: Codex engineering session Repository context: /code Primary implementation period: Phases 006-048 Latest validated evidence run: phase-048-hypothesis-evolution

Abstract

This report documents the design, implementation, and evaluation of an event-focused multi-agent research system that converts public, stakeholder-facing questions about sports events and local economic activity into structured hypotheses, statistical tests, notebooks, evidence reports, and replayable run artifacts. The system began as a schema-only synthetic evidence pipeline and evolved into a governed friends-loop architecture with question generation, semantic-layer exploration, hypothesis validation, matched statistical testing, notebook execution, report rendering, and phase-level regression gates.

The most recent work added an explicit QuestionForum layer, a deterministic tournament of candidate questions, a reflection/reviewer pass, and hypothesis-evolution metadata. These changes were inspired by Google DeepMind's Co-Scientist concept, especially its emphasis on multi-agent hypothesis generation, debate/ranking, reflection, and evolution. The local system is not a biomedical discovery engine and does not claim the breadth, scale, literature grounding, or novelty guarantees of Co-Scientist. It is instead a narrower civic/business evidence workflow: public questions enter from stakeholder personas, analytical machinery maps them into testable hypotheses, and deterministic evidence artifacts report what the current data can and cannot support.

The resulting target architecture separates question demand from analysis execution. Public-facing questions are stored as forum records with persona, source, popularity, priority, rationale, and status. Candidate questions are scored by answerability, novelty, public interest, testability, evidence value, policy relevance, peer-review style reflection, and evolution from prior findings. The system then executes a deterministic friends loop, records all intermediate state, generates a business-readable evidence report and an animated playback report, and validates end-to-end behavior through repeatable 20-turn regression runs.

0. Contributions And Evaluation Summary

This project makes four concrete engineering contributions:

  1. A public-question layer that separates stakeholder demand from statistical execution.
  2. A deterministic governance stack for ranking, reviewing, and evolving candidate questions.
  3. A hypothesis-testing workflow that maps readable questions into explicit H0/H1 contracts, matched tests, diagnostics, notebooks, reports, and telemetry.
  4. A phase-gated validation discipline in which each major governance layer was accepted only after a 20-turn regression run.

The claim is deliberately bounded. The system demonstrates a reproducible event-evidence workflow, not a general scientific discovery engine. It can select, test, report, and replay questions against the repository's event/economic data. It cannot yet provide live literature review, causal identification guarantees, human forum moderation, or model-generated adversarial debate.

1. Motivation

The original project addressed a practical research problem: how can a user ask a fuzzy question about events and local economic impact, and receive a structured, auditable answer rather than a free-form narrative? Early versions focused on synthetic workflows that could validate architecture without requiring real-world truth claims. Later phases moved toward reference data, sports event exposure, local economic measures, semantic query layers, and autonomous discovery loops.

Two tensions drove the architecture:

  1. Public questions must remain readable. Business owners, journalists, policymakers, and citizen journalists ask questions such as "Do big sports crowds actually turn into more local spending?" They do not ask for "block-matched MSA-week revenue_all lift under semantic-layer batch correction."
  2. Statistical evidence must remain explicit. The system still needs formal hypotheses, data contracts, matching rules, diagnostics, multiple-testing handling, and caveats that prevent casual overclaiming.

The project therefore moved toward a layered architecture where public demand drives question framing, while statistical design and execution stay below the surface.

2. Inspiration From Co-Scientist

Google DeepMind describes Co-Scientist as a multi-agent research partner that can generate, debate, refine, and rank hypotheses for complex scientific problems. The public writeup emphasizes a "tournament of ideas" in which agents debate and rank hypotheses, and broader materials describe generation, reflection, ranking, evolution, and meta-review agents coordinated around research goals. Source: https://deepmind.google/blog/co-scientist-a-multi-agent-ai-partner-to-accelerate-research/.

This project borrowed four architectural ideas from that pattern:

  • Generate: produce many candidate questions and hypotheses.
  • Debate/rank: compare candidates rather than only scoring them independently.
  • Reflect: critique whether a question is misleading, weakened by prior evidence, or answerable.
  • Evolve: split, combine, strengthen, or carry forward hypotheses based on memory.

The local implementation intentionally differs from Co-Scientist in important ways:

  • It is deterministic and repository-local, not a large-scale model-orchestrated biomedical discovery system.
  • It is grounded primarily in event/economic reference data, semantic SQL outputs, notebooks, and statistical test runners, not a broad scientific literature corpus.
  • It optimizes for auditable evidence production and reproducible phase gates, not open-ended discovery at thousands-of-hypotheses scale.
  • It exposes public questions and reports for civic/business audiences, not lab proposals for wet-lab validation.

The Co-Scientist lesson was not "copy the product." The lesson was that hypothesis work improves when candidate ideas are generated, debated, reviewed, evolved, and then validated against evidence rather than accepted from a single-pass prompt.

3. Research Questions

This engineering thesis asks:

  1. Can a stakeholder-facing question layer improve the readability and usefulness of autonomous data-science questions?
  2. Can deterministic tournament, reflection, and evolution passes approximate useful multi-agent reasoning without live model calls?
  3. Can the system keep public question framing separate from formal hypothesis testing?
  4. Can the full workflow remain regression-testable through repeated 20-turn runs?

The implemented answer is yes, with caveats. The current architecture successfully carries public question metadata through 20-turn runs, produces reports, and records tournament/reflection/evolution metadata. The caveat is that the tournament, reflection, and evolution layers are deterministic heuristics rather than deep model-based scientific reasoning.

3.1 Methodology And Evaluation Protocol

The project uses engineering evaluation rather than a human-subject or wet-lab methodology. Each phase introduces one major capability, then runs the friends loop for 20 turns to test whether the capability survives realistic system orchestration.

The evaluation unit is a completed workflow turn. A turn is successful when the system can select a candidate, route it correctly, produce the expected evidence artifacts, and persist the governance metadata introduced by the phase. Phase summaries therefore check:

  • completed workflow count
  • early-stop status
  • workflow-task misroutes into statistical testing
  • selected-candidate metadata coverage
  • report and telemetry generation
  • notebook and evidence artifact presence when applicable

This protocol is stricter than a unit test alone because it exercises the candidate policy, routing, statistical runner, reporting code, and persistence layer together. It is still not a substitute for external validity testing against new domains or human evaluation of question quality.

4. System Overview

The core system is a friends loop. Five friend roles participate through markdown-backed role specs:

  • Spark: exploratory data scientist.
  • Skeptic: identification-minded reviewer.
  • Mapper: semantic/data lineage connector.
  • Moderator: ranking and selection lead.
  • DataAgent: notebook and artifact producer.

The loop runs turn by turn. Each turn proposes candidates, enriches them with metadata, scores and filters them, selects one, maps it to a workflow, generates evidence artifacts, appends notebook knowledge, writes wiki-style records, and persists telemetry.

5. QuestionForum Store

The QuestionForum store is the layer above the friends. It represents the forum of questions that business owners, economists, journalists, citizen journalists, politicians, and policymakers might ask. The current store is JSON-backed at data/question_forum/questions.json and loaded by app/question_forum.py.

Each record includes:

  • question_id
  • kind
  • persona
  • question
  • rationale
  • priority
  • popularity
  • source_url
  • status
  • tags

Allowed statuses are:

  • proposed
  • selected
  • tested
  • answered
  • needs-review

The key design decision is that public questions are first-class data, not string constants hidden in candidate policy. A forum record can ask:

Do big sports crowds actually turn into more local spending?

The analysis layer may internally map that to:

H0: event-exposed MSA-weeks have no higher mean revenue_all than matched no-game MSA-week controls.
H1: event-exposed MSA-weeks have higher mean revenue_all than matched no-game MSA-week controls.
Design: matched MSA-week controls.

The public question remains the headline, while the formal hypothesis remains the test contract.

6. Candidate Tournament

Before the tournament layer, candidate scoring was a weighted scalar based on answerability, interestingness, novelty, continuity, business readability, provenance, family coverage, and freshness. That was useful but opaque. It did not resemble debate.

The tournament layer in app/question_tournament.py adds deterministic pairwise comparison. It computes five component scores:

  • public interest
  • novelty
  • testability
  • evidence value
  • policy/business relevance

Candidates then participate in pairwise comparisons. Each candidate receives:

  • tournament_rank
  • tournament_score
  • tournament_wins
  • tournament_losses
  • tournament_component_scores
  • tournament_transcript
flowchart LR A[Candidate A] --> C{Pairwise Debate} B[Candidate B] --> C C --> D[Component Scores] D --> E[Winner] D --> F[Transcript] E --> G[Tournament Rank] F --> H[Candidate Audit Trail]

The transcript is not a live LLM argument. It is a compact deterministic explanation of why one candidate won the comparison. That is an intentional intermediate step. It gives auditability now and leaves a clean seam for future live debate agents.

7. Reflection And Review

The reflection pass in app/question_reflection.py asks three questions for every candidate:

  1. What would make this question misleading?
  2. What prior evidence would weaken this?
  3. Is this actually answerable with current data?

The reflection output includes a status:

  • pass
  • needs-review
  • not-answerable

Reflection has two jobs. First, it adds review metadata to the candidate. Second, it adjusts scores so misleading or weakly answerable questions become less likely to be selected. Workflow-like items can be marked not answerable and filtered.

flowchart TD A[Candidate Question] --> B[Misleading Risk Check] A --> C[Weakening Evidence Check] A --> D[Data Answerability Check] B --> E[Reflection Payload] C --> E D --> E E --> F{Status} F -->|pass| G[Small Score Bonus] F -->|needs-review| H[Score Penalty] F -->|not-answerable| I[Filter From Selection]

This layer was added after the system began to produce questions that were technically useful but too convoluted. The reflection pass is a guard against two failure modes:

  • overclaiming exploratory observational evidence
  • asking questions that current data cannot answer cleanly

8. Hypothesis Evolution

The evolution layer in app/hypothesis_evolution.py records how a candidate relates to prior memory. It can assign four action types:

  • split
  • combine
  • strengthen
  • carry_forward

Examples:

  • Split: a broad question combines spending and merchant activity.
  • Combine: prior questions in the same family should be compared rather than interpreted separately.
  • Strengthen: weak, null, or not-testable evidence should be reframed into a cleaner variant.
  • Carry forward: failed or null tests should influence the next candidate instead of being forgotten.

The current implementation records evolution metadata and applies small score nudges. It does not yet generate full new natural-language variants from a model. That is a reasonable next step, but the metadata foundation is now present.

10. Hypothesis Testing Workflow

The hypothesis-testing workflow has several routes, depending on the question kind and available data.

10.1 Public Question To Formal Hypothesis

Candidate questions are classified as statistical hypotheses, EDA questions, or workflow tasks. Valid statistical hypotheses carry:

  • H0
  • H1
  • population
  • unit of analysis
  • exposure
  • comparison group
  • outcome
  • expected direction
  • test family
  • alpha
  • decision rule

EDA questions can be graduated into formal hypotheses when there is enough structure. Workflow tasks are kept out of the statistical queue.

The conversion step is intentionally lossy in one direction only. A public question may be broad, but a formal hypothesis must name the population, exposure, comparison group, outcome, and unit of analysis. If the system cannot name those fields, it should route the item to EDA or workflow handling rather than pretending it has a statistical design.

10.2 City-Week Matched Tests

For city-week questions, the system uses matched city-week controls. A treated city-week is compared with non-game or comparable weeks, with diagnostics such as:

  • matched pair count
  • eligible treated rows
  • eligible control rows
  • same-month share
  • same-block share
  • median population distance
  • match quality

The statistical runner returns decisions such as:

  • reject_h0
  • fail_to_reject_h0
  • not_testable
  • inconclusive

10.3 MSA-Week Matched Tests

Semantic SQL can produce MSA-week leads. These are mapped to MSA-week hypotheses and tested with matched MSA-week controls. The design prefers same compact MSA blocks and same-month matching, with county population-weight sum as a size proxy.

10.4 Multiple Testing And Evidence Promotion

After turns complete, statistical results receive multiple-testing correction. Evidence promotion assesses robustness based on:

  • adjusted decision
  • replication status
  • diagnostics
  • notebook validation state
  • caveats
  • design limitations

Reports intentionally frame findings as exploratory observational evidence. Rejecting H0 means the observed result is unlikely under the null model and assumptions. It does not prove that sports events caused economic changes.

10.5 Interpretation Discipline

The system uses statistical decisions as evidence-routing signals, not as final truth labels. A result can be promoted only with its diagnostics, caveats, and design limits attached. The report layer should preserve at least four distinctions:

  • statistically detectable association versus causal effect
  • public-interest question versus formally testable hypothesis
  • exploratory screening versus confirmatory analysis
  • null or weak evidence versus an unimportant question

These distinctions are product requirements. Without them, the same automation that makes the system useful would also make it easy to overstate observational findings.

13. Implementation Anatomy

The implemented system is not a single agent prompt. It is a deterministic software workflow with agent-shaped roles around it. That distinction is central to the engineering result. The friend roles supply perspective, tone, and division of labor, but the durable behavior lives in typed records, candidate policy, statistical runners, notebooks, telemetry events, and report generators.

At a high level, a run enters through app/main.py, which exposes the friends-loop command-line surface. The caller supplies the turn count, output run id, notebook directory, data-pull mode, agent mode, and notebook execution backend. The default notebook directory is common-analysis/notebooks; the default execution backend is lightweight. A full run then calls the friends loop, which prepares the notebook workspace, reads prior wiki state, reads notebook knowledge, proposes candidates, ranks them, selects a candidate, executes the appropriate workflow, writes evidence artifacts, creates notebooks, appends wiki entries, records telemetry, applies post-run multiple-testing correction, and writes the final session package.

The core implementation boundary is:

  • app/friends_loop.py: turn orchestration, friend conversation, candidate selection, workflow execution, telemetry events, and session construction.
  • app/friends_candidate_policy.py: candidate templates, score features, readability rewrites, forum integration, tournament/reflection/evolution integration, and suppression of workflow tasks from statistical routing.
  • app/question_forum.py: JSON-backed public question records and status validation.
  • app/question_tournament.py: deterministic pairwise comparison and component scoring.
  • app/question_reflection.py: misleading-risk, weakening-evidence, and answerability checks.
  • app/hypothesis_evolution.py: split, combine, strengthen, and carry-forward metadata.
  • app/statistical_hypothesis.py: validation and classification of candidate text into statistical hypotheses, EDA questions, and workflow tasks.
  • app/experimental_design.py and statistical runner modules: selection and execution of matched designs.
  • app/data_exploration_agent.py: notebook workspace, notebooks, Markdown exports, and wiki writes.
  • app/notebook_execution.py: notebook execution and validation.
  • app/notebook_knowledge_base.py: durable memory extracted from notebooks.
  • app/friends_session.py: final session JSON, Markdown, playback telemetry, business report, and discovery decision report.

This structure emerged because early versions put too much responsibility into the candidate prompt and too little into testable code. As soon as the system had to run 20-turn regressions, hidden prompt behavior became a liability. The implementation moved toward small deterministic modules with JSON-compatible payloads, so a phase could prove that a new layer was present on every selected candidate and did not break routing, reporting, or notebook production.

13.1 Turn Lifecycle

Each turn is a small research cycle. The loop begins by reading the current wiki and notebook knowledge base. That prior memory matters because the system is not meant to ask isolated questions forever. It should know what it has already tested, what generated weak evidence, which notebooks exist, and which semantic slots are becoming repetitive.

The turn then proposes a board of candidates. These candidates may come from static templates, forum questions, semantic-layer outputs, EDA seeds, notebook knowledge seeds, or lineage/evolution pressure from prior turns. Each candidate is a structured object with fields such as kind, family, metric_focus, population_scope, comparison_lens, question, rationale, freshness_claim, source_notebook, source_cell, and score metadata. This gives the loop a way to reason about duplicate questions and question families without parsing prose every time.

Candidate scoring then applies several layers. The older scoring layer ranks answerability, interestingness, novelty, continuity, business readability, provenance, family coverage, and freshness. The newer governance stack enriches that with forum metadata, tournament rank, reflection status, and evolution action. The Moderator selects one candidate. Rejected candidates remain visible in the session so a reviewer can see what the system chose not to pursue.

After selection, the system classifies the candidate. A candidate may be a statistical hypothesis, an EDA question, or a workflow task. This step was introduced because the system had a recurring failure mode: artifact-management questions, notebook-reuse questions, and lineage questions could look important but should not be sent to the statistical test runner. Workflow tasks now remain workflow tasks. EDA questions can produce descriptive artifacts and later graduate into hypotheses. Statistical hypotheses must carry enough formal structure to support H0/H1 execution.

The selected workflow then produces a result summary. Depending on the kind, the workflow may use reference city-week data, MSA-week semantic results, matched controls, batch screens, descriptive EDA artifacts, or fallback summaries. The result summary is not just a sentence. It includes metric, value, top city or MSA when applicable, caveat, claim strength, methodology card, reasoning trace, formal hypothesis, experimental design, statistical test result, diagnostics, and notebook validation contract when available.

DataAgent then writes the notebook. The notebook embeds the selected candidate, the result payload, methodology cards, experimental design, statistical test output, diagnostics, EDA artifacts, and a validation contract. The notebook is executed, exported to Markdown, indexed, and promoted into the notebook knowledge base. Finally, the friend panel writes wiki entries and the loop records telemetry for everything that happened.

The turn lifecycle can be summarized as:

  1. Read wiki and notebook knowledge.
  2. Propose candidate questions.
  3. Enrich candidates with forum, tournament, reflection, and evolution metadata.
  4. Score and rank the board.
  5. Select one candidate.
  6. Classify the candidate as hypothesis, EDA, or workflow.
  7. Execute the chosen workflow.
  8. Write notebook and Markdown export.
  9. Append wiki entries and notebook knowledge.
  10. Emit telemetry events.
  11. Persist turn payload into the final session.

The important implementation decision is that each stage leaves evidence behind. There is no invisible "agent thought" that the system must trust. The thesis treats the visible artifacts as the research object.

13.2 Friend Roles As Prompted Interfaces

The friends are Spark, Skeptic, Mapper, Moderator, and DataAgent. Their names make the workflow easier to understand, but the implementation uses them as interfaces rather than autonomous personalities. Spark is responsible for idea generation and curiosity. Skeptic protects methodology and claim boundaries. Mapper links the question to data, semantic slots, and prior notebooks. Moderator owns selection and board discipline. DataAgent owns durable artifacts.

This division gives the system a human-readable model of collaboration. It also gives telemetry a vocabulary. A playback event from Mapper means something different from a playback event from Moderator. A thesis reader can inspect a run and see which role read knowledge, which role ranked the board, and which role wrote the notebook.

The implementation intentionally avoids relying on private chain-of-thought. Notebook and wiki exposition is public-facing reasoning: a concise explanation of rationale, assumptions, checks, and caveats. For example, app/data_exploration_agent.py writes an "Agent Exposition" section that says why Spark found the question interesting, what Skeptic's main caveat was, which semantic slot Mapper linked, why Moderator selected the candidate, and what reusable result DataAgent produced. That is not hidden model deliberation. It is an auditable record designed for future agents and human reviewers.

This is one of the central lessons of the project. Multi-agent architecture should not mean "five opaque agents talk and we hope for the best." In this system, the agent roles are useful because they organize responsibilities and artifacts. The durable behavior still comes from explicit records and deterministic checks.

13.3 Candidate Policy And Question Governance

The candidate policy is the heart of the question layer. It started as a board of hardcoded candidates and became a governance pipeline. The policy now has to solve four problems:

  1. Generate or retrieve plausible questions.
  2. Keep the questions readable for stakeholders.
  3. Prevent duplicate or low-value turns.
  4. Route only valid statistical hypotheses into statistical tests.

The QuestionForum layer addressed the first two problems. Instead of burying public-facing questions inside code, records live in data/question_forum/questions.json. Each record has question text, persona, rationale, priority, popularity, source URL, tags, and status. This makes the public demand layer visible and editable. It also gives the system a way to represent the user idea that business people, economists, journalists, citizen journalists, politicians, and policymakers might post popular questions to a forum that guides the friends.

The tournament layer addressed the third problem. Simple scalar scoring was useful but did not resemble debate. The tournament compares candidates pairwise and scores public interest, novelty, testability, evidence value, and policy/business relevance. The output includes wins, losses, component scores, rank, and a transcript. The transcript is deterministic, but it gives maintainers something to inspect when a candidate wins.

The reflection layer addressed the fourth problem. It asks what would make the question misleading, what prior evidence would weaken it, and whether the current data can answer it. The output can pass, require review, or mark the candidate not answerable. This is especially important because the system often finds statistically significant associations in observational data. A popular question can be important and still be misleading if the report implies causality.

The evolution layer gives the system a memory-aware question posture. Broad questions can be split. Related questions can be combined. Weak questions can be strengthened. Failed or null tests can be carried forward as constraints on future claims. The current implementation records metadata and adjusts scores; it does not yet rewrite rich variants using a live model. That limitation is acceptable for the present phase because it gives the rest of the system a stable contract for future evolution.

13.4 Statistical Routing And Claim Boundaries

One of the hardest implementation challenges was keeping different kinds of work separate. The system frequently wants to ask questions such as:

  • Which city should we inspect next?
  • Which notebook should reuse the current result?
  • Which result needs a robustness check?
  • Which major-market spending lift should we test?

Only the last question is clearly a statistical hypothesis. The others may be important but they are workflow or EDA questions. Earlier versions risked sending workflow tasks into statistical runners, which created malformed claims and noisy reports. The fix was not merely a better prompt. The fix was a classification and validation layer.

The statistical-hypothesis validator requires a formal structure: H0, H1, population, unit of analysis, exposure, comparison group, outcome, expected direction, test family, alpha, and decision rule. If those fields are missing, the candidate should not be treated as a statistical claim. This pushed the architecture toward an important discipline: public questions can be broad, but formal tests must be narrow.

The report language then reinforces the boundary. A reject_h0 decision is not written as "the event caused spending." It is written as an exploratory observational result under a recorded design. That design includes matching criteria, diagnostics, p-values, adjusted decisions, and caveats. This prevents the public layer from exaggerating the statistical layer.

The system therefore has two kinds of readability. Public readability makes the question understandable. Methodological readability makes the evidence auditable. A good thesis must emphasize both.

14. Challenges Faced

The project faced several recurring engineering and research challenges. They are worth documenting because they shaped the final architecture more than any initial plan did.

14.1 Public Questions Became Too Technical

The most visible challenge was that the questions became hard to follow. The system had internal handles such as semantic_msa_batch_supported_positive, matched_msa_week_controls, and revenue_all. Those handles are useful inside code. They are terrible as public questions. When such handles leaked into reports, the output looked like an internal data-science notebook rather than a tool for business owners or civic stakeholders.

The first attempted fix was wording cleanup. That helped some cases, but it was not enough. The deeper issue was architectural. If the only source of questions is candidate policy code, then the system naturally speaks in implementation terms. The QuestionForum layer fixed this by creating a layer above the friends. It made stakeholder questions first-class data with personas, rationale, popularity, source, and status.

The lesson is that "make the wording better" is too small a fix when the wrong layer owns the wording. Public-question framing needs its own data model.

14.2 Statistical Workflows Were Easy To Overclaim

The system repeatedly found significant p-values in exploratory observational data. That is exciting, but dangerous. Sports events are not randomly assigned. Local spending can vary with seasonality, city size, tourism, weather, concurrent events, team performance, macroeconomic factors, and data coverage. A matched city-week design can reduce some confounding, but it cannot make causal claims by itself.

This forced the system to add claim boundaries at multiple layers. The methodology cards state the design. The statistical result records diagnostics. The evidence promotion layer keeps results from becoming publishable findings merely because a p-value is small. The report evaluator downgrades claim strength when notebooks are only scaffolded or lightweight-executed. The business report uses cautious language.

The challenge was not only statistical. It was product-level. A user reading an HTML report may not distinguish "reject H0 under an exploratory matched design" from "the event caused revenue." The system has to make the safer interpretation the default.

14.3 Notebook Execution Was Initially Too Weak

Early notebooks were useful as scaffolds, but scaffolded notebooks are not evidence. A notebook that is written but never executed can contain stale code, missing imports, inconsistent values, or an untested interpretation. The project therefore added notebook execution and Markdown export.

The first execution path is a lightweight executor. It walks code cells, executes Python in a constrained namespace, captures output, and saves execution metadata back into the notebook. This is enough for deterministic generated notebooks that use simple Python and pandas-style summaries. It is not a full Jupyter kernel. The code also supports an nbclient backend when full Jupyter execution is needed.

The system then added validation contracts. A turn notebook can embed a contract derived from the statistical runner. The notebook recomputes the observed contract from the embedded result payload and asserts that it matches. This is a practical bridge between "notebook as presentation" and "notebook as evidence object." It does not make the analysis perfect, but it prevents the notebook from silently drifting away from the runner output.

14.4 Memory Could Become Either Too Sparse Or Too Noisy

The friends loop needs memory. Without memory, it repeats questions, forgets null results, and fails to build a research trajectory. With too much unstructured memory, it drowns in logs. The project solved this by creating two memory surfaces: a Markdown wiki and a structured notebook knowledge base.

The wiki is human and agent readable. It includes index.md, log.md, question-board.md, decision-records.md, caveats.md, semantic-map.md, and findings.md. These files are append-oriented and readable. The knowledge base is structured JSON plus Markdown. It extracts notebook cells, status, validation metadata, hypothesis seeds, semantic slots, and latest entries. Candidate policy can then use this memory to seed follow-up candidates.

The challenge is balance. If the system only writes prose, it becomes hard to query. If it only writes JSON, it becomes hard for humans and LLMs to use. The dual wiki/JSON pattern is a pragmatic compromise.

14.5 Observability Had To Serve Humans And Tests

The first kind of observability was developer observability: can tests tell whether a phase regressed? The second kind was product observability: can a human replay what the agents did? The third kind was research observability: can a future analyst inspect how a claim was produced?

The final system writes all three. phase_regression_summary.json supports phase gates. friends_loop_telemetry.json and ui/index.html support playback. friends_loop_session.json, notebooks, Markdown exports, and discovery decision reports support research inspection. This became a major thesis point: observability is not an add-on. In an autonomous research system, observability is part of the research method.

14.6 Multi-Agent Inspiration Needed Deterministic Grounding

Co-Scientist inspired the architecture, but copying a high-scale multi-agent scientific system was neither possible nor appropriate. The repository needed something testable in local runs. Deterministic governance layers were therefore a deliberate compromise. They are less expressive than live debate agents, but they are inspectable, repeatable, and regression-testable.

This created a productive tension. The advisor-style critique is that deterministic transcripts are not true debate. The student response is that true debate without reproducibility would not have solved the current engineering problem. The present architecture creates stable interfaces where live agents can be added later. It also creates a baseline that future live agents must beat.

15. LLM Wiki And Memory Mechanisms

The project uses an LLM wiki pattern: generated notebooks and their Markdown exports become durable, readable memory for future agent turns. The phrase "wiki" is important because the system is not simply logging events. It is building an analysis workspace that future agents can read, cite, and extend.

The workspace is created by ensure_notebook_workspace in app/data_exploration_agent.py. It creates:

  • SCHEMA.md
  • index.md
  • log.md
  • question-board.md
  • decision-records.md
  • caveats.md
  • semantic-map.md
  • findings.md

SCHEMA.md documents the operating conventions. Raw data remains immutable. Generated notebooks compile reusable analysis. Markdown exports serve as agent-readable wiki pages. index.md catalogs artifacts. log.md records append-only activity. Agents should prefer the Markdown export for reading and the .ipynb file when execution metadata is needed. Semantic SQL is governed through whitelisted views and SELECT-only constraints.

This design supports three readers:

  1. Humans who need to inspect the project.
  2. LLM agents that need compact textual memory.
  3. Deterministic code that needs structured metadata.

The wiki files are intentionally simple. They avoid a database dependency and keep the knowledge surface visible in versioned files. The tradeoff is that append-only Markdown can become long. That is why the system also maintains notebook-knowledge.json.

15.1 Notebook Knowledge Base

The notebook knowledge base is maintained by app/notebook_knowledge_base.py. It reads notebooks, extracts cells, detects execution status, reads validation metadata, records semantic slots, and writes two artifacts:

  • notebook-knowledge.json
  • notebook-knowledge.md

The JSON artifact is for code. The Markdown artifact is for agents and humans. A knowledge entry includes notebook path, Markdown path, selected candidate, result summary, cells, notebook status, notebook validation, validation contract, semantic slot, and hypothesis seeds. The system also records a summary with entry count, latest notebook, latest seed question, and latest source cell.

This gives the next turn a memory query. Before ranking candidates, the loop reads the knowledge base and can say how many entries exist, which notebook is latest, and which seed is available. Candidate policy can then create follow-up questions such as:

Which broader business question should reuse the current result as a starting point?

or:

Which external replication or stronger causal-design notebook should we create before treating the lift as durable?

Those questions are not random. They are generated because prior notebooks created seeds, and the system can cite the source notebook and source cell.

15.2 Shared Wiki Entries

After each turn, append_friend_wiki_entries writes role-specific and artifact-specific summaries to the wiki. The entries include the selected question, candidate slot, notebook path, Markdown path, caveats, decision records, and findings. The writing pattern is append-oriented. Old entries are not rewritten because the project wants an audit trail, not a mutable narrative.

The wiki also records caveats separately from findings. That separation matters. In research systems, caveats are often lost when a result is summarized. Here, caveats are first-class memory. If a later turn asks a related question, the caveat can come forward through the knowledge base and candidate policy.

15.3 Wiki As A Coordination Protocol

The wiki is not only documentation. It is a coordination protocol between runs. A future agent session can inspect common-analysis/notebooks/index.md and notebook-knowledge.md to understand what the system already did. This matters in a Codex workflow where sessions can be compacted, interrupted, or resumed. Durable files are more reliable than chat memory.

The wiki also supports governance. If a candidate claims novelty, the system can compare it against prior semantic slots and notebook seeds. If a candidate claims a finding is reusable, the system can check whether the notebook exists and whether it executed. If a candidate should be weakened by prior evidence, reflection can cite that prior evidence.

15.4 Limitations Of The Wiki Mechanism

The wiki is useful but incomplete. It does not yet provide semantic search over all notebook exports. It does not summarize long histories into hierarchical review articles. It does not enforce a formal ontology of findings. It can accumulate stale entries if later runs supersede earlier evidence. It is also possible for generated prose to be too verbose for efficient LLM use.

The next version should add an indexed memory layer. That layer could compute embeddings over Markdown exports, track supersession relationships, and generate periodic review notes. However, the current wiki is the correct foundation: it is simple, transparent, and already tied to executable notebooks.

16. Notebook Execution And Evidence Objects

Notebook execution is one of the most important implementation details because it changes notebooks from display files into evidence objects. A notebook in this system is not only a chart or narrative. It is a structured artifact that embeds the selected candidate, result payload, methodology, formal hypothesis, test result, diagnostics, caveats, and validation contract.

The notebook writer lives in app/data_exploration_agent.py. For every selected turn, write_turn_notebook creates a file named:

turn-<number>-<candidate-kind>.ipynb

It also writes a Markdown export beside it. The notebook starts with human-readable sections and includes code cells that render structured payloads as tables. If the result has a methodology card, the notebook includes it. If the result has a reasoning trace, the notebook includes it. If the result has an experimental design, it includes a design table. If the result has a formal hypothesis, it includes H0/H1 and related fields. If it has a statistical test result, it includes the executed test output and the validation contract.

The validation contract is particularly important. A simplified version of the generated notebook cell is:

validation_contract = result_summary.get("notebook_validation_contract", {})
observed_contract = {
    "schema_version": validation_contract.get("schema_version"),
    "kind": selected_candidate.get("kind"),
    "semantic_slot": "|".join(
        str(selected_candidate.get(key, ""))
        for key in ["family", "metric_focus", "population_scope", "comparison_lens"]
    ),
    "metric": result_summary.get("metric"),
    "value": result_summary.get("value"),
    "claim_strength": result_summary.get("claim_strength"),
    "statistical_test_result": {
        key: result_summary.get("statistical_test_result", {}).get(key)
        for key in validation_contract.get("statistical_test_result", {})
    },
    "data_snapshot_hashes": validation_contract.get("data_snapshot_hashes", {}),
}
validation_passed = observed_contract == validation_contract
assert validation_passed

This cell does not prove the statistical design is correct, but it proves that the notebook-visible result matches the runner-visible result for the checked fields. That is a meaningful guardrail. It prevents a report from displaying one value while the notebook silently contains another.

16.1 Lightweight Execution

The lightweight execution backend in app/notebook_execution.py executes generated Python code cells without launching a full Jupyter kernel. It maintains a namespace, executes statements, evaluates final expressions where possible, and records display output. This is enough for current generated notebooks because their cells are simple and deterministic.

The benefit is speed and low dependency cost. A 20-turn run can execute many notebooks without requiring a heavy Jupyter environment. The cost is fidelity. Lightweight execution is not identical to full notebook execution. It may not support every Jupyter behavior, magic command, display hook, or kernel state nuance. The report evaluator therefore caps notebook maturity for lightweight-executed notebooks. Lightweight execution is better than scaffolding, but it is not full validation.

16.2 Nbclient Execution

The code also supports an nbclient backend. When selected, the payload is converted to a notebook object and executed with nbclient. This is closer to real Jupyter execution and should be used when the notebook becomes a serious validation artifact. The system already has the command-line option:

--notebook-execution-backend nbclient

The thesis should be clear that current default runs generally use lightweight execution unless explicitly configured otherwise. This is a maturity distinction, not a defect. The architecture supports the stronger backend, but the phase gates were primarily about end-to-end governance metadata and report generation.

16.3 Sample Notebook: Question Bank

The first notebook in a run is the question bank:

000-question-bank.ipynb

It records the initial ranked board available to the friend panel. A representative structure is:

# Upfront Question Bank

This notebook records the initial ranked board available to the data-scientist friend panel.

## Panel Exposition

Skeptic protects causal and statistical boundaries.
Mapper links candidates to semantic slots and prior artifacts.
Moderator ranks the board.
DataAgent turns selected questions into executed notebooks and Markdown wiki pages.

The question bank notebook is useful because it records the space of alternatives at the beginning of a run. Without it, only selected questions would be durable. A thesis-level system should preserve rejected options because selection is part of the research process.

16.5 Sample Notebook: Multiple-Testing Correction

After all turns complete, the system writes:

999-multiple-testing-corrections.ipynb

The distinction is subtle but important. A per-turn notebook is an immutable record of what happened during the turn. The correction notebook is a run-level synthesis. Mutating earlier notebooks with later correction data would make the audit trail harder to reason about.

16.6 Notebook Status And Maturity

The report evaluator recognizes notebook status. A notebook can be absent, scaffolded, lightweight-executed, nbclient-executed, validated, or published. The system uses this status when judging actionability. A scaffolded notebook cannot support a validated finding. A lightweight-executed notebook improves evidence but is capped. A validated notebook can support stronger claims if methodology and evidence are also mature.

This is a practical mechanism for preventing overclaiming. It forces report maturity to depend on artifact maturity. The thesis should preserve this point: notebook execution is not a side feature. It is part of claim governance.

17. Observability, Telemetry, And Reports

Observability is the difference between an impressive demo and a research system that can be inspected. The project records observability at several levels: turn events, workflow stage audit logs, notebook paths, wiki paths, knowledge summaries, prompt hashes, decision summaries, business reports, playback reports, and phase regression summaries.

17.1 Telemetry Event Stream

The event stream is constructed in _append_turn_telemetry in app/friends_loop.py. For each turn, the system emits events such as:

  • turn.started
  • knowledge.read
  • board.proposed
  • board.ranked
  • semantic.query
  • hypothesis.classified
  • question.submitted
  • workflow.stage
  • result.studied
  • notebook.created
  • wiki.updated
  • knowledge.updated
  • discussion.message
  • memory.seeded

These events provide a timeline. They also provide structured data. For example, a semantic.query event records query id, purpose, SQL, referenced views, row count, row limit, columns, and row preview. A workflow.stage event records stage name, function or agent, input references, output reference, warnings, and timestamp. A notebook.created event records notebook paths. This makes the playback UI more than an animation; it is a structured observability surface.

17.2 Prompt Hashes

The playback telemetry includes friend prompt hashes and evidence prompt hashes. This matters because agent behavior depends on role definitions and prompts. If the prompt changes, a run should not be treated as identical. Hashes let a reviewer compare runs and see whether prompt versions changed. This is the prompt analogue of a code version.

The system does not yet treat prompt hashes as a full reproducibility lock. It does, however, surface them in telemetry and reports, which is the right first step. Future work should pin prompt versions more aggressively and include prompt diff summaries in regression evidence.

17.3 Session JSON

friends_loop_session.json is the canonical run artifact. It includes schema version, run configuration, friends, prompt hashes, turns, notebook workspace, decision summaries, session summary, and telemetry events. Each turn includes selected candidate, rejected candidates, candidate classification, workflow status, packet paths, notebook paths, wiki paths, wiki summary before selection, result summary, and candidate governance metadata.

This file is the best source for programmatic analysis. HTML reports are easier to read, but JSON is easier to test. The 20-turn regression gates rely on the JSON structure because they can count completed workflows, inspect selected candidates, and verify that metadata exists.

17.4 Session Markdown

friends_loop_session.md is a human-readable session summary. It includes the autonomous discovery synthesis, completed turns, selected questions, scores, results, metrics, decisions, packet paths, and rejected candidates. It is not as rich as the JSON, but it is valuable for quick review. A maintainer can open one file and understand what the run did.

17.5 Discovery Decision Summary

discovery_decision_summary.md is a run-level table of decisions. It records requested turns, completed turns, raw decision counts, adjusted decision counts, candidate classifications, EDA turns, statistical-hypothesis turns, workflow turns, lineage turns, and unique adjusted rejected-null hypotheses. It then lists each turn with kind, class, question, H0, H1, adjusted decision, p-value, adjusted p-value, data note, and design note.

This report exists because the project needed a concise way to audit statistical routing. It answers questions such as:

  • Did workflow tasks enter statistical testing?
  • How many hypotheses rejected H0?
  • Were adjusted decisions recorded?
  • Which designs were used?
  • Which questions were only EDA?

That report was essential when the system began generating more complex question types. Without it, routing regressions were hard to see.

17.6 Business Evidence Report

business_evidence_report.html is the stakeholder-facing report. It is meant for users who care about the answer but not about every JSON field. It summarizes questions, findings, caveats, and evidence in a readable single-page format. The business report is not allowed to become a hype layer. It must preserve caveats and avoid causal overclaiming.

The business report is important because the project is not only a backend experiment. The user wanted HTML reports after runs and wanted both business and playback formats produced per run. The implementation now writes both formats automatically. That requirement changed the architecture: report generation became a first-class run artifact rather than an optional post-processing step.

17.7 Playback UI

The playback UI lives under ui/index.html inside each run directory. It renders friends_loop_telemetry.json into a replayable view of the friends loop. The point is not entertainment. The point is traceability. A reviewer can see when the board was proposed, when knowledge was read, when the question was submitted, when the workflow stages fired, when the notebook was created, and when wiki memory was updated.

This is especially useful for multi-agent systems because the failure modes are temporal. A final answer may look wrong because the wrong candidate was selected, because the semantic query returned no rows, because a workflow task was misclassified, because the notebook failed, or because the report overpromoted a weak result. Playback gives the reviewer a time-ordered path to diagnose those failures.

17.8 Phase Regression Summary

The phase regression summary is the delivery gate. For the latest phase, it showed 20 requested turns, 20 completed workflows, no early stop, zero workflow-task statistical misroutes, and full metadata coverage across forum, tournament, reflection, and evolution layers. This is the strongest evidence that the new governance layers were not merely present in code but active in the end-to-end run.

The limitation is that 20 turns is not a statistical proof of correctness. It is an integration and regression standard. It tests orchestration pressure. It can reveal broken paths, missing metadata, misroutes, report failures, and notebook failures. It cannot prove that every future question will be good.

This diagram captures the central observability loop. A turn creates artifacts. Those artifacts create reports. Some artifacts also feed memory. Memory influences the next turn. The system is therefore observable and iterative at the same time.

18. Sample End-To-End Run Narrative

To understand the system, imagine a public forum question:

Do big sports crowds actually turn into more local spending?

The forum record attaches a persona, such as a small-business owner or policymaker. It includes rationale and popularity. Candidate policy reads the record and creates a candidate with a public question and internal semantic slot. Tournament scoring gives it public-interest, novelty, testability, evidence-value, and policy/business relevance scores. Reflection checks whether the question could mislead users into believing the system can prove causality. Evolution checks whether prior spending questions should be combined, split, strengthened, or carried forward.

The selected candidate is then mapped into a formal hypothesis:

H0: Event-exposed locality-weeks have no higher mean revenue_all than matched no-game locality-week controls.
H1: Event-exposed locality-weeks have higher mean revenue_all than matched no-game locality-week controls.

The design router chooses a matched city-week or MSA-week test, depending on available data and question kind. The runner constructs treated and control groups, computes match diagnostics, estimates the difference, records p-values, applies decision rules, and stores caveats. If the run is part of a family, multiple-testing correction is applied after all turns.

DataAgent then writes the notebook. The notebook records the public question, formal hypothesis, design, test result, diagnostics, validation contract, and caveats. The Markdown export makes the result readable to future agents. The wiki is updated. The notebook knowledge base is updated with seeds for later questions.

Telemetry records the turn. The business report summarizes the result cautiously. The playback report shows how the result was produced. The phase regression summary counts whether this turn completed and whether the required metadata was present.

This is the actual contribution of the system. It is not simply that an LLM asks a question. It is that a public question travels through a governed, observable, testable path into evidence artifacts and back into memory.

19. Detailed Lessons Learned

13.1 Question Wording Is A System Design Problem

The project initially tried to improve question wording with string rewrites. That helped but was brittle. The better fix was architectural: make public questions first-class records from a forum layer.

13.2 Analysis Handles Must Not Become Headlines

Internal handles are useful for execution. They should not leak into stakeholder-facing questions. A kind such as semantic_msa_batch_supported_positive is appropriate for code, but the report should say "Which major-market spending lift should we check next?"

13.3 Determinism Is A Strength

Live agent debate would be more expressive, but deterministic tournament, reflection, and evolution are testable. They can be run in CI-like workflows and validated through 20-turn regression gates.

13.4 Statistical Claims Need Guardrails

The system repeatedly returns significant p-values in exploratory observational contexts. Without careful reporting, users may read "reject H0" as causal proof. The architecture now emphasizes caveats, methodology cards, and claim boundaries.

13.5 Notebooks Are Evidence Objects

Notebooks are not just user-facing conveniences. In this system they are durable knowledge artifacts that seed future candidates, record validation contracts, and make analysis reproducible.

13.6 Regression Runs Are Product Tests

The 20-turn run is more than a smoke test. It verifies that candidate selection, hypothesis routing, testing, reports, and telemetry continue to work together.

21. Limitations

The current system has important limitations:

  • The forum is JSON-backed, not a live user-facing product.
  • Tournament debate is deterministic, not true adversarial agent debate.
  • Reflection is heuristic, not literature-grounded peer review.
  • Evolution records actions but does not yet generate rich new hypothesis variants.
  • External literature search is not integrated into the main loop.
  • Matching designs are observational and cannot establish causality.
  • Same-year, COVID-period, placebo, pre-trend, clustered-SE, and block-bootstrap robustness remain incomplete.
  • The system is narrow: sports events and local economic reference data, not general science.

22. Future Work

The next major improvements should be:

  1. Forum UI and submission workflow.
  2. Human voting and moderation.
  3. True model-based tournament debate with deterministic transcript storage.
  4. Literature and web grounding before hypothesis promotion.
  5. Stronger causal designs where data supports them.
  6. Bootstrap or clustered uncertainty for repeated locality-week observations.
  7. Evolution that produces concrete rewritten candidates, not only metadata.
  8. Cross-run dashboard for forum question lifecycle: proposed to selected to tested to answered.

23. Conclusion

This project built a working event-agent research system that turns public questions into auditable evidence artifacts. Its main contribution is not any single statistical test or report. The contribution is the layered architecture: public question demand, multi-agent candidate governance, formal hypothesis testing, notebook-backed evidence, and repeatable 20-turn regression.

The Co-Scientist inspiration clarified the shape of the missing reasoning layers: debate, reflection, ranking, and evolution. The local implementation translated those ideas into deterministic modules that fit the repository's evidence workflow. The result is a pragmatic research assistant for civic and business event questions: not a replacement for scientists, economists, journalists, or policymakers, but a disciplined machine for generating, testing, critiquing, and reporting questions they actually care about.

Appendix B. Sample Notebook Details

This appendix expands the notebook discussion with representative cells and explains why each section exists.

B.1 Notebook Header

A generated turn notebook begins with identity. It should make the selected question visible before any code appears:

This header prevents a common notebook failure: the code may be correct, but the reader cannot tell which research question it answers. The candidate slot also helps future agents decide whether the notebook is relevant.

B.2 Embedded Payload Cell

A notebook needs the selected candidate and result summary as executable data:

selected_candidate = {
    "kind": "semantic_msa_batch_supported_positive",
    "question": "Which major-market spending lift should we check next?",
    "family": "positive_revenue",
    "metric_focus": "revenue_all_difference",

    "comparison_lens": "matched_msa_week_controls",
}

result_summary = {
    "metric": "revenue_all",
    "value": 12345.67,
    "claim_strength": "exploratory_observational",
    "caveat": "Matched observational design; not causal proof.",
}

The actual generated payload is larger. The simplified example shows the principle: the notebook is self-contained enough to render the result without querying hidden runtime state.

B.3 Methodology Card Cell

The methodology card is rendered as data rather than prose alone:

methodology_card = result_summary.get("methodology_card", {})
pd.DataFrame([methodology_card]).T.rename(columns={0: "value"})

This helps reviewers compare methods across notebooks. If one notebook used matched city-week controls and another used MSA-week controls, the method table exposes that difference.

B.4 Formal Hypothesis Cell

The formal hypothesis is also rendered as structured data:

formal_hypothesis = result_summary.get("formal_hypothesis", {})
pd.DataFrame([formal_hypothesis]).T.rename(columns={0: "value"})

Typical fields include H0, H1, population, unit of analysis, exposure, comparison group, outcome, expected direction, test family, alpha, and decision rule. A notebook without these fields may still be useful for EDA, but it should not be treated as a formal hypothesis test.

B.5 Statistical Test Cell

The statistical test result is displayed with the same pattern:

statistical_test_result = result_summary.get("statistical_test_result", {})
pd.DataFrame([statistical_test_result]).T.rename(columns={0: "value"})

This table may include decision, p-value, adjusted p-value, effect estimate, sample counts, matched-pair counts, same-month share, same-block share, and primary design. The purpose is to make the report's claim inspectable.

B.6 Validation Contract Cell

The validation contract is the strongest notebook-specific guardrail:

validation_contract = result_summary.get("notebook_validation_contract", {})
observed_contract = {
    "schema_version": validation_contract.get("schema_version"),
    "kind": selected_candidate.get("kind"),
    "metric": result_summary.get("metric"),
    "value": result_summary.get("value"),
    "claim_strength": result_summary.get("claim_strength"),
}
validation_passed = all(
    observed_contract.get(key) == validation_contract.get(key)
    for key in observed_contract
)
assert validation_passed

The real generated cell also checks selected statistical fields and data snapshot hashes. The simplified version shows the logic. The notebook asserts that its visible payload matches the contract.

B.7 Markdown Export

After execution, the notebook is exported to Markdown. That export is crucial for the LLM wiki because Markdown is cheaper and easier for agents to read than raw .ipynb JSON. The Markdown export preserves headings, prose, code, and outputs. It becomes an agent-readable explanation of the turn.

The Markdown export also improves code review. A reviewer can inspect a notebook result without opening a notebook UI. In a command-line repository workflow, that matters.

The notebook is where candidate governance, statistical design, execution, validation, and wiki memory meet.

Appendix D. Phase Narrative

This appendix gives a narrative of what was accomplished across the project. It is intentionally higher level than the commit history and lower level than the abstract.

D.1 Schema-Only Evidence Pipeline

The earliest work built a schema-only evidence pipeline. The goal was not to prove a real-world sports-economics claim. The goal was to prove that a vague question could be turned into structured artifacts: hypothesis cards, evidence packets, design sketches, and skeptical review. This phase established the idea that an answer should not be a single paragraph. It should be a bundle of claims, assumptions, data references, and review notes.

The main limitation was realism. Synthetic evidence can validate plumbing, but it cannot validate data joins, sample quality, matching, or report interpretation. The project therefore moved toward reference event data.

D.2 Reference Event Data And Local Economic Measures

The next work added reference sports/event data and local economic measures. This gave the system a domain: sports events and local economic activity. The data model made it possible to ask city-week and MSA-week questions. It also introduced new risks. Real data is messier than synthetic examples. Localities differ in population, economic base, event frequency, and data coverage. That pushed the system toward diagnostics and matched comparisons.

D.3 Friends Loop

The friends loop changed the project from a one-shot pipeline into an iterative research system. Spark, Skeptic, Mapper, Moderator, and DataAgent gave the system a collaborative structure. The loop could ask one question, study the result, write memory, and ask another question. This was the point where notebooks and wiki entries became essential. Without memory, the loop would not be research; it would be repeated isolated analysis.

D.4 Shared Wiki

The shared wiki made memory durable. It created a common analysis notebook workspace with schema, index, log, question board, decision records, caveats, semantic map, and findings. This was an important shift from "the agent remembers" to "the repository remembers." The latter is much more reliable in a coding-agent workflow.

D.5 Semantic Layer

The semantic layer gave agents governed SQL access. It constrained queries to whitelisted views, read-only SELECT behavior, and row limits. This let the system explore MSA-week and city-week patterns without letting arbitrary generated SQL mutate data or wander through the repository. The semantic layer also created reusable metric definitions, which reduced the chance that different notebooks would compute inconsistent versions of the same concept.

D.6 Formal Hypothesis Validation

As the loop became more autonomous, it needed stronger routing. Formal hypothesis validation forced candidate questions to declare H0, H1, population, unit, exposure, comparison, outcome, direction, test family, alpha, and decision rule. This reduced ambiguity and made it possible to distinguish statistical hypotheses from EDA and workflow tasks.

D.7 Matched Tests And Multiple Testing

Matched city-week and MSA-week tests gave the system a real statistical execution path. Multiple-testing correction gave the system a family-level guardrail. Together, they turned the project from "descriptive notebook generator" into "exploratory statistical evidence workflow." The claim remained bounded because matching is observational, but the workflow became much more rigorous.

D.8 Business And Playback Reports

The report work made runs inspectable by different audiences. Business reports serve stakeholders. Playback reports serve reviewers and maintainers. Decision summaries serve analysts. Regression summaries serve phase delivery. This multi-report strategy is appropriate because no single artifact can serve every reader.

D.9 QuestionForum, Tournament, Reflection, And Evolution

The latest phases implemented the Co-Scientist-inspired governance stack. QuestionForum moved public demand above the friends. Tournament made selection comparative. Reflection added critique before selection. Evolution made memory part of candidate scoring. The final 20-turn phase showed that all four layers could be active together without breaking workflow execution.

D.10 What Remains Unfinished

The system is still not a finished product. The forum has no user interface. Tournament debate is deterministic. Reflection is heuristic. Evolution records metadata but does not yet generate rich variants. External literature grounding is not integrated. Stronger causal methods remain future work. Notebook execution defaults to lightweight mode. These limitations do not negate the accomplishment. They define the next research agenda.

Appendix F. Rejected Alternatives, Maintenance, And Assessment Criteria

This appendix records design alternatives that were considered implicitly during the project and explains why the implemented path was chosen. This matters because the final architecture can look obvious after the fact. It was not obvious during development.

F.1 Alternative: Put All Agent Logic In Markdown Role Files

One tempting design was to move most agent behavior into Markdown prompts. The repository already uses Markdown role definitions, and the user asked whether most agent logic lives in Markdown files or code. Markdown is valuable because it is inspectable, easy to edit, and understandable to non-programmers. It is the right place for role voice, responsibilities, claim-boundary reminders, and interaction expectations.

The project did not put the main logic there. Candidate scoring, workflow routing, statistical validation, notebook execution, telemetry, and report writing live in code. That decision was necessary because those behaviors need tests. If a workflow task is incorrectly routed into a statistical test, a prompt instruction may reduce the probability of the error, but a code-level classifier and regression check can prevent or detect it deterministically. If every selected candidate must carry tournament metadata, a prompt cannot guarantee that structure; a code path can.

The final division is therefore hybrid. Markdown defines the roles. Code defines the contracts. This is the right architecture for a governed Codex project because it lets non-code artifacts shape behavior without making the system untestable.

F.2 Alternative: Use Live LLM Debate Immediately

Another tempting design was to implement a live multi-agent debate layer immediately. That would have looked more like Co-Scientist. Agents could argue about novelty, public interest, answerability, and evidence value. The output might be more expressive and more surprising than deterministic transcripts.

The project deferred that approach because the system first needed a stable baseline. Live debate introduces nondeterminism, cost, latency, prompt drift, and replay complexity. A regression gate that passes today may fail tomorrow because a model response changes. That does not mean live debate is wrong; it means the deterministic foundation should come first.

The deterministic tournament is therefore an architectural scaffold. It gives the system component scores, pairwise comparisons, ranks, wins, losses, and transcripts. Future live debate can be plugged into the same output contract. When that happens, the deterministic layer can remain as a fallback, benchmark, or prefilter.

F.3 Alternative: Treat Reports As Post-Processing

A simpler implementation would run analyses first and generate reports later only when requested. The project rejected that model. Reports are not optional decoration. In this system, the report is part of the evidence package. If every run is supposed to support stakeholder review and debugging, then every run should write both business and playback HTML.

Making reports part of the run also improved testing. A 20-turn regression can verify that report generation still works after candidate policy changes. If reports were separate, a phase might pass the analytical code but break the user-facing artifact.

F.4 Alternative: Store Memory Only In JSON

The system could have stored all memory in JSON. That would be convenient for code, but poor for human and LLM review. Raw JSON notebooks and event logs are verbose. They are difficult to skim. They also do not naturally support explanatory prose.

The system could also have stored all memory in Markdown. That would be readable, but harder for candidate policy to query. The final design uses both. Markdown is the wiki surface. JSON is the structured memory surface. This is a common pattern in practical agent systems: one representation for machines, one for humans and language models.

F.5 Alternative: Store Notebooks Only, Without Markdown Exports

Keeping only .ipynb files would preserve execution metadata, but it would make agent reading harder. Notebook JSON is noisy and expensive for language models. Markdown exports are compact and readable. They also work well in code review. The project therefore writes both notebook and Markdown versions.

This dual format is especially important because the notebook is both executable evidence and wiki content. The .ipynb file proves execution state. The .md file communicates the result.

F.6 Alternative: Use A Database For The Forum Immediately

A real product forum should eventually use a database. It should support users, moderation, voting, search, provenance, status transitions, and audit logs. The current phase deliberately used a JSON-backed store. That choice was pragmatic. The project needed a real data model for public questions before it needed a deployed forum product.

The JSON store is not a dead end. It defines the record shape: question text, persona, popularity, rationale, source, link, and status. A database-backed implementation can preserve that shape. The thesis should treat the JSON store as a minimum viable persistence layer, not as the final forum architecture.

F.7 Alternative: Promote Significant Results Automatically

The project could have promoted every reject_h0 result as a finding. That would be wrong. Exploratory observational tests, repeated screening, and lightweight notebook execution are not enough for strong claims. The evidence promotion layer and report evaluator prevent that shortcut.

A result must carry method, diagnostics, caveats, notebook status, and multiple-testing context. Even then, the default claim is exploratory. This restraint is central to the system's credibility. The system should be useful because it is disciplined, not because it makes dramatic claims.

F.8 Maintenance Procedure For Future Phases

Future maintainers should treat the thesis architecture as a maintenance contract. A typical change should follow this procedure:

  1. Identify which layer the change affects: forum, candidate governance, hypothesis routing, statistical execution, notebooks, wiki, telemetry, reports, or regression gates.
  2. Add or update focused unit tests for the changed layer.
  3. Run a short targeted workflow if the change affects orchestration.
  4. Run a 20-turn regression if the change affects selected candidates, routing, notebooks, telemetry, or reports.

The system should be assessed against criteria that match its claim. It should not be graded as a general scientific discovery engine, because it does not claim that. It should be assessed as a reproducible event-evidence workflow.

F.10 Ethical And Civic Considerations

The system sits near public decision-making. Even though the current domain is sports events and local spending, the audience could include business owners, journalists, policymakers, and citizens. That creates ethical obligations. A misleading report could influence staffing, policing, public funding, venue policy, or local economic narratives. A poorly framed result could overstate benefits of events or ignore distributional harms.

The system responds to this risk through caveats, reflection, claim boundaries, and observational language. That is necessary but not sufficient. A production version should add human moderation, source transparency, data provenance notes, and explicit uncertainty communication. It should also allow affected communities to contest or contextualize findings. Popular forum questions should guide analysis, but popularity alone should not determine truth or policy priority.

This is another reason the QuestionForum should include persona, rationale, source, and status. A question is not neutral just because it is popular. A policymaker question, business-owner question, and citizen-journalist question may reflect different interests. The system should preserve that context rather than flattening it into a generic prompt.

F.12 How The Thesis Should Be Read

The safest reading of this thesis is as an engineering thesis about building a governed research workflow, not as a domain thesis proving a sports-economics result. The domain matters because it forces real statistical and communication problems, but the primary artifact is the system. The system's quality should be judged by whether it can receive public questions, govern candidate selection, map suitable questions into explicit hypotheses, run appropriate exploratory tests, write notebooks and reports, expose its reasoning trail, and repeat that process without hidden state.

That reading also clarifies the role of the examples. The sample JSON records, notebook cells, and telemetry events are not decorative appendices. They are the practical contract by which the architecture can be reimplemented or audited. If a future version claims to improve the tournament layer, it should still be able to show candidate comparison metadata. If a future version claims to improve notebook maturity, it should still distinguish scaffolded, lightweight-executed, nbclient-executed, and validated notebooks. If a future version claims to improve question quality, it should still preserve the public question separately from the formal hypothesis.

The thesis should also be read as a record of iterative correction. Several important features were added because earlier behavior was unsatisfactory: questions were too convoluted, workflow tasks risked statistical misrouting, notebooks needed execution, reports needed HTML outputs, and observability needed to be visible per run. Those corrections are part of the contribution. They show how an autonomous analysis system can be made more reliable by turning failure modes into explicit contracts.
